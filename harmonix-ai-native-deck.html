Phần I

Agentic Automation Workflows & Skill Chains

Bốn workflow tự động hóa phần việc lặp lại của PM/COO. Toàn bộ giả định đã có "Harmonix Brain" (Phần II) làm nguồn tri thức nền để AI tra cứu trước khi hành động.

WORKFLOW 1

"Idea‑to‑Task" — Ý tưởng thô → PRD → Github Issues

Tự động hóa ~80% công sức viết spec sản phẩm và phân rã task kỹ thuật cho DEV.

01
Input
PM nhập ý tưởng thô qua chat hoặc voice vào Discord/Slack.
02
Soạn PRD
AI truy vấn Harmonix Brain, viết bản nháp PRD hoàn chỉnh trên Notion.
03
Quét rủi ro
Phát hiện rủi ro bảo mật DeFi (rút tiền, oracle, trượt giá), tự điền Security Checklist.
04
Duyệt
PM chỉnh sửa và bấm [Approve] trên Notion.
05
Tạo task
AI gọi Github API tách PRD thành task nhỏ kèm Size: S/M/L, gán người phụ trách.

Skill chains

prd_generation_skill defi_risk_auditor_skill github_issue_builder_skill

Yêu cầu vận hành

  • PM/COO duyệt bản nháp để huấn luyện AI hiểu phong cách công ty
  • CEO/DEV thống nhất quy chuẩn chấm Size: S/M/L
WORKFLOW 2

"Daily Sync & Standup Reporter"

Tự động quét hoạt động Github để cập nhật tiến độ — không cần DEV kéo thẻ Kanban thủ công.

01
Input
Commit, mở/đóng PR, comment của DEV trên Github.
02
Quét & phân tích
AI chạy ngầm quét Github API + Notion Board hằng ngày.
03
Phát hiện nghẽn
Task "In Progress" >3 ngày không commit mới → đánh dấu blocked.
04
Báo cáo
Gửi Daily Sync vào Discord lúc 9h sáng: đã xong / đang làm / cảnh báo nghẽn.
05
Auto Status‑Sync
Tự chuyển thẻ Kanban theo vòng đời branch → PR → merge, một chiều Github → Kanban.
Giải quyết gốc rễ: nhánh mới → In Progress; PR draft → In Review (Draft); PR ready → Waiting Review; PR merge → Done + tự đóng Issue kèm comment Auto‑closed vì PR #X đã merge.

Skill chains

git_activity_fetcher_skill notion_card_tracker_skill standup_generator_skill auto_status_sync_skill

Yêu cầu vận hành (bắt buộc)

  • Tên nhánh có ID task: feature/#102-arbitrum-gas
  • PR ghi từ khóa liên kết: Closes #102
  • Commit message ngắn gọn, rõ nghĩa
WORKFLOW 3

"QA Spec Matcher" — Nghiệm thu tính năng tự động

Tự động hóa khâu click chuột test giao diện và logic trên web testnet.

01
Input
PRD đã duyệt + preview link từ PR.
02
Tạo kịch bản
AI biên soạn test scenarios dưới dạng các bước click chuột.
03
Thực thi
Browser Agent (Playwright ảo) mở trình duyệt, kết nối ví test Web3, chụp screenshot.
04
Báo cáo
So sánh ảnh thực tế với Figma/PRD, gửi Visual Diff cho PM/COO.

Skill chains

test_scenario_compiler_skill browser_action_executor_skill visual_diff_auditor_skill

Yêu cầu vận hành

  • DEV cung cấp link staging/preview ổn định trong PR
  • PM/COO viết rõ Acceptance Criteria trong PRD
WORKFLOW 4

"Github Velocity & Performance Insight"

Tổng hợp tín hiệu khách quan từ Github để hỗ trợ đánh giá năng suất — không thay thế đánh giá con người, chỉ là dữ liệu tham chiếu.

01
Input
Lịch sử commit, PR, review comment, Size: S/M/L/XL trong một chu kỳ.
02
Tính chỉ số
Cycle Time · Review Turnaround · Rework Rate · Estimate Accuracy · Blocked‑time Ratio.
03
Chuẩn hóa
Mọi chỉ số chuẩn hóa theo Size — không so thô số commit/dòng code giữa các DEV.
04
Output
"Github Velocity Report" theo tuần/tháng — biểu đồ xu hướng, không phải điểm số một lần.
Nguyên tắc bắt buộc: dữ liệu chẩn đoán, không phải KPI trực tiếp; không dùng làm căn cứ duy nhất để thưởng/phạt; DEV luôn xem được báo cáo của chính mình.

Skill chains

github_velocity_analyzer_skill estimate_accuracy_tracker_skill review_turnaround_tracker_skill

Yêu cầu vận hành

  • DEV tuân thủ quy tắc branch/PR/task‑size ở Workflow 2
  • PM/COO dùng báo cáo để hỗ trợ 1‑1 hằng tháng, không chấm điểm tự động
Phần II

Kịch bản xây dựng Harmonix Brain (Context Layer)

Để mọi Agent ở Phần I không bị "ảo giác", Harmonix cần một kho tri thức chung — Vector Database hợp nhất Notion và Github, chạy trên Server Mac Mini của công ty.

INPUT
Notion API + Github API/Webhooks
Tài liệu sản phẩm, code, PR, Issues, comments
↓
PROCESSING
LlamaIndex / LangChain
Chunking + embedding, gắn metadata
↓
STORAGE
Pgvector hoặc Pinecone
Vector DB lưu tri thức sống của công ty
↓
INTERFACE
Claude 3.5 Sonnet + Bot Discord/Telegram
Trả lời có dẫn chứng, đồng bộ ngược khi sai

5 bước triển khai

BƯỚC 1
Dọn dẹp dữ liệu nguồn (Data Curation)
Gắn Status cho mọi trang Notion: [Canonical/Verified] · [Active Draft] · [Draft]/[Archived] (bỏ qua, không nạp vào AI).
BƯỚC 2
Phân loại nguồn dữ liệu Github
Xếp hạng độ tin cậy từng loại nguồn (bảng bên dưới) + metadata bắt buộc: repo, pr_number/issue_number, branch, author, merged_at, labels, status.
BƯỚC 3
Context Engineering
Metadata‑rich chunking + Hierarchy of Truth: code trên main > ghi chú PR/Issue đã closed > tài liệu Notion.
BƯỚC 4
Đồng bộ real-time qua Github Webhook
Merge PR / Close Issue / comment gắn #spec, #note → cập nhật ngay Vector DB.
BƯỚC 5
Bot Discord/Telegram + Feedback Loop
Tag @Harmonix-Brain để hỏi; nếu trả lời sai do Notion cũ, gõ @Harmonix-Brain sync [link] để cập nhật ngay.

Độ tin cậy nguồn dữ liệu Github

Không phải mọi tín hiệu từ Github đều đáng tin như nhau — AI cần biết cái nào là quyết định cuối, cái nào chỉ là thảo luận.

NguồnĐộ tin cậyCách xử lý
Code + docstring trên mainCao nhấtChỉ index README, docs/, comment giải thích logic
PR description + comment (merged)Cao"Decision log" kỹ thuật, giữ PR number/tác giả làm metadata
Issue đã đóng, kèm comment cuốiTrung bình–caoComment cuối trước khi đóng = kết luận thật
Issue đang mởThấpGắn nhãn [Đang thảo luận] khi trích dẫn
Commit messageRất thấpChỉ trace lịch sử, không giải thích logic
Wiki / CHANGELOGTrung bìnhXử lý như Notion, bắt buộc last_updated

Xung đột Notion vs Github — "cái nào là sự thật?"

So sánh last_updated (Notion) với merged_at (PR gần nhất): PR mới hơn → Github thắng, cảnh báo Notion lỗi thời; Notion mới hơn → Notion thắng, nhắc DEV code chưa khớp spec; không rõ thứ tự → AI không tự chọn phe, gắn nhãn [Xung đột chưa rõ ràng] để người quyết định.

Code chạy trước, tài liệu chưa viết xong (Documentation Lag)

Tình huống bình thường ở startup tốc độ cao — không phải lỗi, nhưng cần quy trình rõ để AI không "mù thông tin".

PHÁT HIỆN
PR merged không gắn notion_url → đánh dấu [Chưa có tài liệu — Code‑First].
TẠM THỜI
Agent tự soạn Stub Doc từ PR description + code diff, gắn nhãn [Auto‑generated / Chờ PM xác nhận]; luôn cảnh báo khi trích dẫn.
ESCALATE
Sau 7 ngày chưa có tài liệu chính thức → nhắc PM qua Discord (Notify — Tier 1).
CHỐT
PM viết xong → gắn #supersedes-stub, Stub Doc chuyển [Archived]. Tạo Stub = Tier 0; chốt tài liệu = Tier 2 (PM duyệt).
Phần III

Cổng phê duyệt (Human‑in‑the‑loop)

Hành động nào AI được tự làm, hành động nào bắt buộc phải có người duyệt? Nguyên tắc hybrid autonomy: tự động hóa việc thường quy, chặn lại ở quyết định tác động cao. Approval card gửi qua bot Discord (Phần II, Bước 5) kèm nút [Approve]/[Reject], sau đó đồng bộ ngược Notion/Github.

Bốn cấp độ tự chủ

TIER 0 · AUTO
AI thực hiện và log lại, không cần duyệt trước.
Quét git activity, soạn Daily Sync report, cập nhật thẻ Kanban.
TIER 1 · NOTIFY
Thực hiện ngay, báo real‑time; có cửa sổ "veto" ngắn.
Cảnh báo nghẽn (blocked task), báo cáo NAV hằng ngày.
TIER 2 · APPROVE‑1
AI đề xuất, cần 1 người duyệt trước khi thực thi.
Duyệt PRD (Workflow 1 — Issues tự tách sau đó), đổi UX/feature flag.
TIER 3 · APPROVE‑MULTISIG
Cần ≥2 người duyệt độc lập — tài chính/pháp lý lớn, khó đảo ngược.
Rebalance vượt ngưỡng, đổi tham số chiến lược, whitelist token, chi reward lớn.

Phân loại hành động thay đổi lớn theo bộ phận

Bộ phậnHành động ví dụTierNgười duyệt
Trading / Vault OpsRebalance vượt ngưỡng, đổi tham số chiến lược, whitelist token, tạm dừng vault3Trưởng Trading + COO
FinanceĐổi công thức fee/yield, reward vượt ngưỡng, NAV bất thường3Finance lead + COO
EngineeringDeploy/nâng cấp contract, đổi quyền admin/multisig, CI/CD production3Eng lead + COO*
ProductDuyệt PRD, roadmap chính, UX diện rộng, pricing hiển thị2PM
Growth / MarketingAirdrop/point lớn, thông báo chính sách, partnership mới2→3*PM/Growth lead (+COO)
DataĐổi mô hình điểm/reward, xóa/sửa dữ liệu lịch sử2–3Data lead + COO*
Operations / CompanyChính sách nội bộ, quyền truy cập hệ thống, vendor mới2COO

* khi ảnh hưởng fund hoặc có cam kết tài chính.

Vai trò chi tiết

Approver Tier 2

Product Manager

  • Duyệt mặc định phạm vi sản phẩm: PRD, roadmap, feature flag, UX diện rộng, pricing hiển thị
  • Không tự duyệt gì chạm tiền/smart contract/dữ liệu tài chính — luôn escalate lên Tier 3
  • Approval card cần: mục tiêu, phạm vi ảnh hưởng, khả năng rollback, rủi ro từ defi_risk_auditor_skill, diff trước/sau
SLA: phản hồi trong 4 giờ làm việc → quá hạn tự escalate lên COO
Approver Tier 3

Chief Operating Officer

  • Bắt buộc cho mọi Tier 3, không ngoại lệ — kể cả khi trưởng bộ phận đã duyệt
  • Chịu trách nhiệm cuối: tài chính, pháp lý/compliance, danh tiếng, quyền multisig on‑chain
  • Approval card cần thêm: tác động USD/%TVL, rủi ro pháp lý gắn cờ sẵn, ai duyệt Tier 2 trước đó, kịch bản xấu nhất + rollback
  • Có thể override Tier 2 (một chiều); không ai override được "từ chối" của COO trừ founder/board
SLA: phản hồi trong 2 giờ hành chính + kênh khẩn/on‑call riêng

Audit trail & escalation

01Log toàn bộ — kể cả Tier 0: agent đề xuất, người duyệt, thời gian, approval card gốc, kết quả thực thi, vào Harmonix Brain.
02Fail‑safe khi quá SLA — tự escalate PM → COO → founder; hành động tạm dừng, không mặc định cho qua.
03Review định kỳ — rà lại quyết định Tier 2/3 hàng tuần/tháng để tránh "rubber‑stamp" và tinh chỉnh ngưỡng.
